# MatchTune — backend (MVP прототип)

Бэкенд для дейтинг-приложения, где матчинг строится на основе музыкальной совместимости по публичным плейлистам.

## Что уже работает

- Адаптер Spotify (Client Credentials Flow — без логина пользователя, только серверные ключи приложения)
- Адаптер Яндекс Музыки (неофициальный API, без логина — только публичные плейлисты; см. риски ниже)
- Парсинг ссылки на публичный плейлист → список треков с жанрами
- Построение музыкального профиля (взвешенные векторы жанров и артистов)
- Расчёт совместимости между двумя профилями (косинусное сходство + коэффициент Жаккара)
- Архитектура через адаптер-паттерн — легко добавить SoundCloud позже

## Поддерживаемые ссылки на плейлисты

| Провайдер | Формат ссылки |
|---|---|
| Spotify | `open.spotify.com/playlist/{id}` |
| Яндекс Музыка | `music.yandex.ru/users/{логин}/playlists/{id}` |

**Риск по Яндекс Музыке:** официального API нет, адаптер обращается к внутреннему API веб/мобильного клиента по структуре, которую задокументировало сообщество. Формат ответа может измениться без предупреждения — если адаптер начнёт падать с ошибкой парсинга, вероятно, structure поменялась и адаптер нужно обновить. Часть эндпоинтов может со временем начать требовать `YANDEX_MUSIC_TOKEN` даже для публичных данных — переменная в `.env` уже предусмотрена.

## Установка и запуск

```bash
npm install
cp .env.example .env
# впишите в .env свои SPOTIFY_CLIENT_ID и SPOTIFY_CLIENT_SECRET
npm start
```

Ключи Spotify получаются бесплатно за 5 минут на https://developer.spotify.com/dashboard — создайте приложение, тип "Web API".

Откройте **http://localhost:3001** в браузере — там уже готовый интерфейс: вставляете ссылку на публичный плейлист Spotify → получаете реальный список треков → можете снять галочки с ненужных треков → ниже сразу пересчитывается рейтинг совместимости с 13 демо-профилями.

## База данных (опционально, но рекомендуется)

Без БД всё тоже работает — просто каждый запрос идёт в Spotify заново. С БД треки кэшируются на 24 часа, плюс появляются таблицы под профили/лайки/мэтчи для следующих шагов.

1. Поднимите PostgreSQL (локально или в облаке — подойдёт бесплатный тариф Render/Neon/Timeweb)
2. Впишите `DATABASE_URL` в `.env`
3. Накатите схему:
   ```bash
   psql "$DATABASE_URL" -f db/schema.sql
   ```
4. Перезапустите `npm start` — в ответе `/api/profile` появится поле `fromCache: true/false`, показывающее, откуда взялись треки

Если `DATABASE_URL` не задан — приложение тихо работает без кэша, ничего не ломается.

## Аутентификация

Простая регистрация по email/паролю с JWT-токеном.

1. Впишите `JWT_SECRET` в `.env` (любая случайная строка, например из `openssl rand -hex 32`)
2. Требуется настроенная БД (см. выше) — пользователи хранятся в таблице `users`

**`POST /api/auth/register`**
```json
// запрос
{ "email": "user@example.com", "password": "min8chars", "name": "Аня", "age": 26 }
// ответ
{ "token": "eyJ...", "user": { "id": 1, "email": "...", "name": "Аня", ... } }
```

**`POST /api/auth/login`**
```json
{ "email": "user@example.com", "password": "min8chars" }
```

**`GET /api/auth/me`** — требует заголовок `Authorization: Bearer <token>`, возвращает текущего пользователя.

Токен живёт 30 дней. Пароли хранятся только в виде bcrypt-хэша, в открытом виде нигде не сохраняются.

## Сохранение музыкального профиля

**`POST /api/profile/save`** (требует `Authorization: Bearer <token>`)
```json
{ "genreWeights": { "инди-рок": 1, "дрим-поп": 0.7 }, "artistWeights": { "...": 1 } }
```
Именно этот профиль (посчитанный на фронтенде из выбранных пользователем треков) участвует в подборе кандидатов ниже.

## Пул кандидатов

**`GET /api/candidates?city=Москва&minAge=22&maxAge=30`** (требует авторизации)

Возвращает пользователей с заполненным `music_profile`, отсортированных по убыванию совместимости. Исключает вас самих и тех, кого вы уже лайкнули. Фильтры `city`/`minAge`/`maxAge` — опциональны.

```json
{
  "candidates": [
    { "id": 5, "name": "Марина", "age": 27, "city": "Москва", "score": 62, "sharedGenres": ["инди-рок"], ... }
  ]
}
```

## Лайки и мэтчи

**`POST /api/likes`** — `{ "toUserId": 5 }`. Если собеседник уже лайкнул вас в ответ — сразу создаётся мэтч (`matched: true`).

**`GET /api/matches`** — список ваших мэтчей.

## Чат между мэтчами

**`GET /api/matches/:id/messages`** — история переписки (только для участников мэтча).

**`POST /api/matches/:id/messages`** — `{ "content": "Привет!" }`.

Сейчас это обычный REST — сообщения нужно запрашивать заново (polling), realtime-уведомлений нет. Для живого чата на следующем шаге стоит добавить WebSocket (например, Socket.io) поверх этих же таблиц.

## API

### `POST /api/profile`
Строит музыкальный профиль по ссылке на плейлист.

```json
// запрос
{ "playlistUrl": "https://open.spotify.com/playlist/37i9dQZF1DXcBWIGoYBM5M" }

// ответ
{
  "provider": "spotify",
  "trackCount": 42,
  "tracks": [{ "title": "...", "artist": "..." }],
  "profile": {
    "genreWeights": { "indie rock": 1, "dream pop": 0.7 },
    "artistWeights": { "...": 1 },
    "trackCount": 42
  }
}
```

### `POST /api/match`
Считает совместимость между двумя профилями (полученными из `/api/profile`).

```json
// запрос
{ "profileA": { ...профиль пользователя 1 }, "profileB": { ...профиль пользователя 2 } }

// ответ
{
  "score": 38,
  "breakdown": { "genreCosine": 0.47, "genreJaccard": 0.25, "artistJaccard": 0.33 },
  "sharedGenres": ["indie rock"],
  "sharedArtists": ["Artist A"]
}
```

## Логика проверена

Модули `buildProfile.js` и `similarity.js` протестированы на моковых данных — построение профиля и расчёт совместимости работают корректно (без реального похода в Spotify API, для проверки чистой логики).
